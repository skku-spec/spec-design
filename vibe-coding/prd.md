# PRD 작성법
<!-- new -->

> **PRD는 AI한테 주는 "작업 지시서"입니다.** 기획 없이 바로 만들면 시간도 오래 걸리고, 처음부터 다시 만들어야 할 수도 있습니다. 4가지 질문에 답하고, AI한테 PRD를 만들어달라고 요청하면 한 번에 정확한 결과물을 얻을 수 있습니다.
>
> 이 챕터에서 배우실 내용은 다음과 같습니다.
> 1. PRD가 무엇이고 왜 필요한지
> 2. 아이디어를 정리하는 4가지 질문
> 3. PRD 필수 항목과 작성법
> 4. AI에게 PRD 요청하는 프롬프트 템플릿
> 5. PRD를 Claude Code에 넘기는 방법

## PRD란?

PRD는 **Product Requirements Document**(제품 요구사항 문서)의 줄임말입니다. 쉽게 말하면, **"이렇게 만들어줘"를 정리한 문서**입니다.

### 비유로 이해하기

| 카페에서 주문할 때 | Claude한테 부탁할 때 |
|-------------|----------------|
| "커피 주세요" → 뭔 커피요? | "예약 시스템 만들어줘" → 뭔 예약이요? |
| "아이스 아메리카노, 샷 추가, 덜 달게" → 바로 이해 | "이 PRD 보고 만들어줘" → 바로 이해 |

### PRD가 있으면 뭐가 좋은가요?

1. **Claude가 한 번에 이해합니다.** — PRD 없이 "예약 시스템 만들어줘"라고 하면 뭘 예약하는 건지 모릅니다. PRD가 있으면 정확히 이해하고 만듭니다.
2. **빠뜨리는 게 없습니다.** — 미리 화면 목록, 기능 목록을 정리해두니까 빠뜨릴 일이 없습니다.
3. **나중에 수정하기 쉽습니다.** — "3번 화면의 버튼 색깔 바꿔줘"처럼 명확하게 지시할 수 있습니다.

## 기획 없이 만들면 생기는 일

기획 없이 바로 만들기 시작하면 이런 일이 벌어집니다.

"예약 시스템 만들어줘" → Claude가 만들어줌 → "어? 날짜 선택이 없네?" → "시간 선택도 없네?" → "관리자 페이지도 필요한데?" → 이것저것 추가하다가 **구조가 꼬입니다.**

이게 왜 문제냐면:

1. **시간 낭비** — 같은 것을 두 번 이상 만들게 됩니다.
2. **비용 낭비** — Claude 사용량이 2배 이상 늘어납니다.
3. **멘탈 붕괴** — "이게 맞나?" 계속 헷갈리게 됩니다.

***PRD가 있으면 화면 목록과 기능 목록을 미리 정해두고, "이 PRD 보고 만들어줘" 한 마디로 한 번에 정확하게 만들 수 있습니다.***

## 아이디어 정리하기: 4가지 질문

머릿속에 "이런 거 만들고 싶다"가 있는데 어떻게 정리해야 할지 모르겠다면, **딱 4가지 질문**에만 답해보세요.

1. **누가** 쓰나요? (타겟)
2. **왜** 쓰나요? (목적/문제)
3. **언제** 쓰나요? (사용 상황)
4. **어떻게** 쓰나요? (핵심 기능)

### 예시: 헬스장 1:1 코칭 예약

| 질문 | 답변 |
|------|------|
| 누가 쓰나요? | 내 코칭을 받고 싶은 회원들 |
| 왜 쓰나요? | 카톡으로 일일이 조율하기 귀찮아서 |
| 언제 쓰나요? | 코칭 받고 싶을 때 / 예약 확인할 때 |
| 어떻게 쓰나요? | 날짜 선택 → 시간 선택 → 예약 완료 |

이렇게 답하면 PRD의 뼈대가 완성됩니다.

- **타겟**: 1:1 코칭 희망자
- **목적**: 예약 과정 자동화
- **핵심 기능**: 날짜/시간 선택, 예약 확인

## PRD 필수 항목 4가지

4가지 질문에 답했으면, 이제 PRD 형식으로 정리합니다.

1. **목적** — 이 서비스가 왜 필요한가? (1~2줄)
2. **타겟** — 누가 쓰는가? (1줄)
3. **핵심 기능** — 뭘 할 수 있는가? (3~5개)
4. **화면 구성** — 어떤 페이지들이 필요한가?

### 각 항목 작성법

**1. 목적 (1~2줄)**

"___을/를 해결하기 위한 서비스" 형식으로 씁니다.

- 좋은 예: "카톡으로 예약 조율하는 번거로움을 없애기 위해"
- 나쁜 예: "좋은 서비스를 만들기 위해" (너무 모호함)

**2. 타겟 (1줄)**

"___하는 사람들" 형식으로 씁니다.

- 좋은 예: "내 코칭을 받고 싶은 사람들"
- 나쁜 예: "모든 사람" (너무 넓음)

**3. 핵심 기능 (3~5개)**

"사용자가 할 수 있는 것"을 적습니다. 너무 많으면 3개로 줄입니다.

- 좋은 예: 날짜 선택, 시간 선택, 예약자 정보 입력, 예약 완료 확인
- 나쁜 예: 예쁜 디자인, 빠른 속도 (기능이 아님)

**4. 화면 구성**

필요한 페이지를 나열하고, 각 페이지에 뭐가 있는지 간단히 적습니다.

1. 메인 페이지 — 서비스 소개 + 예약하기 버튼
2. 예약 페이지 — 날짜/시간 선택 + 정보 입력
3. 완료 페이지 — 예약 완료 메시지
4. 관리자 페이지 — 예약 목록 확인

## AI에게 PRD 요청하기

4가지 질문에 답했으면, AI한테 PRD를 만들어달라고 할 차례입니다. Claude, ChatGPT, Gemini 뭘 써도 됩니다.

### 복붙용 프롬프트 템플릿

아래 템플릿을 복사해서 빈칸만 채워 AI에게 보내세요.

```
다음 정보를 바탕으로 PRD(제품 요구사항 문서)를 작성해줘.

[서비스 목적]
(왜 이 서비스를 만드는지 1~2줄)

[타겟 사용자]
(누가 쓰는지 1줄)

[핵심 기능]
- 기능 1
- 기능 2
- 기능 3

[필요한 페이지]
1. 페이지 1 - 설명
2. 페이지 2 - 설명
3. 페이지 3 - 설명

요청사항:
- 한국어로 작성해줘
- 각 화면별로 어떤 UI 요소(버튼, 입력창, 목록 등)가 필요한지 구체적으로 설명해줘
- 사용자가 어떤 순서로 서비스를 이용하는지 흐름(User Flow)을 정리해줘
- 핵심 기능이 3~5개를 넘지 않도록 범위를 좁혀줘
- 첫 버전에서 빠져도 되는 기능이 있으면 "향후 추가 기능"으로 분리해줘
```

## AI가 만든 PRD 검토하기

AI가 PRD를 만들어줬다고 바로 쓰면 안 됩니다. **직접 읽어보면서 점검하는 습관**이 필요합니다.

실제 서비스를 상상하면서 읽어보세요. "사용자가 이 버튼을 누르면 어떻게 되지?", "여기서 에러가 나면 어떻게 되지?" — 이렇게 머릿속으로 시뮬레이션을 돌려보는 겁니다.

### 혼자 검토하기 어려우면?

AI한테 이 **2가지만** 물어보세요.

1. **"예외 처리 정책이 빠진 부분 있어?"** — 입력값이 잘못됐을 때, 인터넷 끊겼을 때, 버튼을 연타로 눌렀을 때 등
2. **"논리적으로 이상한 부분 있어?"** — A 화면에서 B로 가는 버튼이 없다거나, 기능 설명이 서로 충돌하거나

이 2가지 질문으로 대부분의 허점을 잡을 수 있습니다.

### 자주 발생하는 문제 & 해결법

| 문제 | 해결 방법 |
|------|---------|
| AI가 너무 많은 기능을 추가함 | "첫 버전은 핵심 기능만 남기고 나머지는 삭제해줘. 결제, 알림은 나중에 추가할 거야." |
| 설명이 너무 모호함 | "'사용자 친화적인 UI'가 구체적으로 뭔지 설명해줘. 어떤 버튼이 어디에 있는지." |
| 내가 원했던 기능이 빠짐 | "관리자가 예약 목록을 확인하고 취소할 수 있는 페이지 추가해줘" |

## Claude Code에 PRD 넘기기

### PRD를 .md 파일로 저장하기

.md(마크다운) 파일은 **텍스트만 있는 간단한 파일**입니다. Claude Code가 바로 읽을 수 있어서, 프로젝트 폴더에 넣어두면 "PRD.md 읽고 만들어줘"라고 말할 수 있습니다.

**저장 방법**

1. AI가 만들어준 PRD를 전체 복사합니다.
2. 메모장(Windows) 또는 텍스트 편집기(Mac)에 붙여넣습니다.
3. 파일 이름을 `PRD.md`로 저장합니다.
4. **저장 위치는 반드시 프로젝트 폴더 안**이어야 합니다.

```
my-project/
├── PRD.md  ← 여기에 있어야 Claude Code가 읽을 수 있음
```

AI에게 "다운로드 받을 수 있는 md 파일로 보내줘"라고 요청해도 됩니다.

### Claude Code에서 실행하기

1. 터미널에서 프로젝트 폴더로 이동합니다. (`cd my-project`)
2. `ls` 명령어로 PRD.md 파일이 보이는지 확인합니다.
3. `claude`를 입력해서 Claude Code를 시작합니다.
4. 아래와 같이 요청합니다.

```
PRD.md 파일 읽고 그대로 만들어줘.
```

더 구체적으로 말하고 싶다면:

```
지금 폴더에 있는 PRD.md를 읽고,
거기 적힌 대로 웹앱을 만들어줘.
- 첫 버전에서는 PRD에 적힌 핵심 기능만 구현해줘.
- 화면 구조는 PRD의 [필요한 페이지] 항목에 맞춰줘.
```

Claude가 코드를 작성하기 시작하면, 중간중간 질문에 `Y` 또는 `Enter`로 승인하면 됩니다. 완성되면 `npm run dev`로 결과를 확인합니다.

### PRD대로 안 만들었을 때

- **PRD랑 다르게 만들었을 때**: "PRD에는 할 일이 3개만 입력 가능하다고 했는데, 지금은 제한이 없어. PRD 다시 확인하고 고쳐줘."
- **너무 복잡하게 만들었을 때**: "더 간단하게 만들어줘. PRD에 있는 기능만 넣어줘."
- **에러가 날 때**: 에러 메시지를 복사해서 Claude에게 보여주세요.

## AI 기획의 한계

AI가 PRD도 만들어주고 코드도 만들어주지만, **AI에게도 한계가 있습니다.**

| AI가 잘하는 것 | 내가 결정해야 하는 것 |
|------------|--------------|
| 문서 정리하기 | 이 서비스가 진짜 필요한가? |
| 빠진 항목 찾기 | 누구한테 팔 건가? |
| 기술 추천하기 | 가장 중요한 기능이 뭔가? |
| 코드 작성하기 | 이 가격이 맞는가? |
| 에러 고치기 | 이 이름이 좋은가? |

***핵심: AI가 만들어준 PRD는 초안입니다. 그대로 쓰는 게 아니라, 검토하고 수정해서 쓰는 습관을 가져야 합니다.***

## PRD 체크리스트

### 작성 전

- 4가지 질문에 답했나요? (누가/왜/언제/어떻게)
- 이 서비스를 30초 안에 설명할 수 있나요?
- 첫 버전에 들어갈 기능 범위를 정했나요? (3~5개)

### 작성 후

- 목적이 1문장으로 명확한가요?
- 타겟이 구체적인가요? ("모든 사람"이면 안 됨)
- 핵심 기능이 3~5개인가요?
- 화면 목록이 있나요?
- 내가 원했던 기능이 다 들어갔나요?

### Claude Code에 넘기기 전

- .md 파일로 저장했나요?
- **프로젝트 폴더 안에** 저장했나요?
- `ls` 명령어로 PRD.md가 보이나요?
